# 앵클 수동 테스트 체크리스트

이 문서는 `dev` 기준 현재 서비스에서 실제로 밟을 수 있는 주요 사용자/운영 시나리오를 배치 단위로 정리한 실행 체크리스트다.

목표는 두 가지다.

1. 사용자는 이 문서를 보고 하나씩 실제 서비스를 끝까지 돌린다.
2. 개발자는 같은 배치의 코드 경로를 병렬로 점검해 잡버그 후보와 상태 불일치를 찾는다.

## 사용 방법

- 순서는 반드시 `Batch 0 -> Batch 9`로 진행한다.
- `P0 blocker`가 나오면 다음 배치로 넘어가지 않는다.
- 실패하면 아래 형식으로 기록한다.
  - 시나리오 ID
  - 실패한 단계 번호
  - 실제 화면 문구 또는 API 응답 코드
  - 스크린샷 1장 또는 콘솔/네트워크 핵심 한 줄
- 각 배치 종료 후 정리한다.
  - 통과 시나리오
  - 실패 시나리오
  - 잔액/상태/알림/DB 정합성 확인 여부

## 사전 준비

### 테스트 계정

- `U0` 비로그인 상태 브라우저
- `U1` 신규 카카오 계정
- `U2` 전화 인증/가입정보/온보딩을 모두 끝낸 카카오 계정
- `U3` 신규 이메일 가입용 계정
- `U4` 가입 완료된 이메일 계정
- `U5` 캐시를 충분히 가진 계정
- `U6` 매치 신청 완료 이력이 있는 계정
- `U7` 캐시 환불 대기 상태 계정
- `U8` 탈퇴 대기 상태 계정
- `A1` 관리자 계정

### 테스트 매치 상태

- `M1` 신청 가능한 일반 오픈 매치
- `M2` 정원 임박 오픈 매치
- `M3` 정원 마감 매치
- `M4` 시작 2시간 이내 매치
- `M5` 시작 시간이 지난 매치
- `M6` 참가자 미달 전날 예외 테스트용 매치
- `M7` 참가자 미달 당일 예외 테스트용 매치
- `M8` 강수 안내 발송 테스트용 매치
- `M9` 강수 변동 안내 테스트용 매치
- `M10` 관리자 취소 테스트용 매치

### 공통 확인 항목

- 새로고침 후 상태 유지
- 뒤로가기/재진입
- 중복 클릭 또는 빠른 연속 클릭
- My Page 반영
- 관리자 화면 반영
- 알림 발송 여부
- 잔액/환불/신청 상태 정합성

## Batch 0. 테스트 환경 준비

### 코드 감사 포인트

- 인증 진입 게이트: `src/app/auth/continue/route.ts`
- 회원 설정 게이트: `src/lib/member-access.ts`, `src/lib/profile-onboarding.ts`
- 주요 금전 API: `src/app/api/cash/charge-orders/route.ts`, `src/app/api/cash/refund-requests/route.ts`
- 매치 신청/취소 API: `src/app/api/matches/[matchId]/applications/route.ts`, `src/app/api/matches/[matchId]/applications/me/route.ts`
- 탈퇴 API: `src/app/api/account/withdrawal/route.ts`

### 체크리스트

- [ ] ENV가 모두 준비돼 있다.
  - Supabase
  - Toss
  - Solapi
- [ ] 관리자 계정으로 `/admin` 진입이 가능하다.
- [ ] 일반 계정으로 `/mypage`, `/mypage/cash`, `/mypage/settings` 진입이 가능하다.
- [ ] 테스트 매치 `M1 ~ M10` 중 실제로 필요한 상태가 준비돼 있다.
- [ ] 알림 발송 확인용 휴대폰 번호를 확보했다.

## Batch 1. 인증, 회원가입, 전화 인증, 온보딩

### 코드 감사 포인트

- `auth/callback -> auth/continue -> signup/complete -> welcome` 순서가 항상 유지되는지
- 이메일 회원가입이 휴대폰 인증 쿠키에 의존하므로 쿠키 만료/유실에 취약하지 않은지
- `phoneVerificationRequired`, `signupProfileRequired`, `onboardingRequired` 세 게이트가 서로 충돌하지 않는지
- 단일 세션 쿠키가 기존 로그인 세션을 예상대로 밀어내는지

### 시나리오

#### AUTH-KAKAO-01 신규 카카오 첫 로그인

- 사전조건: `U1`
- 단계:
  1. `/login` 진입
  2. 카카오 로그인 진행
  3. 서비스 복귀
- 기대결과:
  - 전화 인증 또는 가입정보 입력 또는 온보딩 화면으로 이동한다.
  - 곧바로 홈으로 빠지지 않는다.

#### AUTH-KAKAO-02 카카오 로그인 후 원래 페이지 복귀

- 사전조건: 비로그인 상태, `/match/[slug]/apply` 또는 `/cash/charge` 직접 접근 링크 준비
- 단계:
  1. 보호된 페이지 진입
  2. 로그인으로 이동
  3. 카카오 로그인 완료
- 기대결과:
  - 중간 게이트를 통과한 뒤 원래 페이지로 복귀한다.

#### AUTH-KAKAO-03 온보딩 완료 계정 재로그인

- 사전조건: `U2`
- 단계:
  1. 로그아웃
  2. 카카오 로그인 재진입
- 기대결과:
  - `verify-phone`, `signup/complete`, `welcome`으로 다시 가지 않는다.

#### AUTH-EMAIL-01 이메일 회원가입 정상 흐름

- 사전조건: `U3`
- 단계:
  1. `/signup` 진입
  2. 이메일/비밀번호/이름/생년월일/성별 입력
  3. 휴대폰 인증 완료
  4. 약관 동의 후 회원가입
- 기대결과:
  - 자동 로그인 후 `auth/continue`를 거쳐 필요한 후속 화면으로 이동한다.
  - 동일 세션에서 바로 서비스 사용 가능하다.

#### AUTH-EMAIL-02 이메일 회원가입 중 휴대폰 인증 없이 제출

- 사전조건: `U3`
- 단계:
  1. 회원가입 폼 입력
  2. 휴대폰 인증 없이 제출 시도
- 기대결과:
  - 제출이 막히거나 인라인 에러가 나온다.

#### AUTH-EMAIL-03 이메일 중복 가입

- 사전조건: 이미 가입된 이메일 존재
- 단계:
  1. 같은 이메일로 회원가입 시도
- 기대결과:
  - 충돌 메시지가 보인다.
  - 기존 계정을 덮어쓰지 않는다.

#### AUTH-EMAIL-04 동일 휴대폰 재가입

- 사전조건: 이미 다른 계정에 연결된 번호 존재
- 단계:
  1. 새 이메일로 회원가입
  2. 기존 번호로 휴대폰 인증 후 제출 시도
- 기대결과:
  - 휴대폰 중복 가입 오류가 나온다.

#### AUTH-EMAIL-05 이메일 로그인 정상

- 사전조건: `U4`
- 단계:
  1. `/login/email` 진입
  2. 올바른 이메일/비밀번호 입력
- 기대결과:
  - `auth/continue`를 거쳐 정상 진입한다.

#### AUTH-EMAIL-06 이메일 로그인 비밀번호 오류

- 사전조건: `U4`
- 단계:
  1. 잘못된 비밀번호로 로그인 시도
- 기대결과:
  - 인라인 에러가 뜬다.
  - 세션이 생기지 않는다.

#### AUTH-PHONE-01 휴대폰 인증번호 재전송

- 사전조건: 회원가입 또는 활성화용 휴대폰 인증 화면
- 단계:
  1. 인증번호 발송
  2. 즉시 재전송 시도
  3. 쿨다운 후 재전송 시도
- 기대결과:
  - 쿨다운 중 재전송 제한이 적용된다.
  - 쿨다운 후 재전송 가능하다.

#### AUTH-PHONE-02 인증번호 오류/만료

- 사전조건: 휴대폰 인증 발송 완료
- 단계:
  1. 틀린 번호 입력
  2. 만료 후 다시 입력
- 기대결과:
  - 오류/만료 메시지가 적절히 나온다.
  - 필요 시 재발송을 유도한다.

#### AUTH-WELCOME-01 임시 레벨 선택 완료

- 사전조건: 온보딩 필요 상태 계정
- 단계:
  1. `/welcome` 진입
  2. 임시 레벨과 선호값 선택
  3. 완료 후 홈 또는 원래 화면으로 이동
- 기대결과:
  - 재로그인 시 `/welcome`이 다시 뜨지 않는다.

#### AUTH-SESSION-01 한쪽에서 로그인하면 다른 세션 밀림

- 사전조건: 같은 계정으로 두 브라우저 준비
- 단계:
  1. 브라우저 A 로그인
  2. 브라우저 B 로그인
  3. 브라우저 A에서 보호 페이지 새로고침
- 기대결과:
  - 단일 세션 정책에 맞게 한쪽이 밀리거나 재로그인을 요구한다.

## Batch 3. 캐시 충전

### 코드 감사 포인트

- 주문 생성 실패, Toss SDK 실행 실패, 결제 취소, 성공 URL 재진입 시 중복 승인 방지
- `PHONE_VERIFICATION_REQUIRED`, `ONBOARDING_REQUIRED`가 충전 API에서도 동일하게 막히는지
- 성공 후 잔액/거래내역/알림이 모두 일치하는지

### 시나리오

#### CASH-CHARGE-01 5,000원 충전 성공

- 사전조건: 전화 인증과 온보딩을 끝낸 계정
- 단계:
  1. `/cash/charge` 진입
  2. 5,000원 선택
  3. 결제수단 선택
  4. 결제 성공
- 기대결과:
  - 성공 페이지가 뜬다.
  - 잔액이 증가한다.
  - 캐시 내역에 `charge`가 찍힌다.
  - 충전 알림이 발송된다.

#### CASH-CHARGE-02 다른 금액 패키지 선택

- 사전조건: 동일
- 단계:
  1. 10,000원
  2. 30,000원
  3. 각 버튼 선택 상태 확인
- 기대결과:
  - 선택된 금액만 강조된다.
  - 결제 버튼 문구가 선택 금액과 맞는다.

#### CASH-CHARGE-03 결제수단 변경

- 사전조건: 동일
- 단계:
  1. 카드
  2. 계좌이체
  3. 카카오페이/네이버페이/토스페이
- 기대결과:
  - 선택된 수단만 강조된다.
  - Toss 결제창이 해당 방식으로 열린다.

#### CASH-CHARGE-04 결제창에서 취소

- 사전조건: 동일
- 단계:
  1. 결제창 진입
  2. 사용자 취소
- 기대결과:
  - 현재 충전 페이지로 복귀한다.
  - 주문만 남고 잔액은 증가하지 않는다.

#### CASH-CHARGE-05 결제 실패 페이지

- 사전조건: 실패를 유도할 수 있는 결제 수단 준비
- 단계:
  1. 결제 실패 유도
- 기대결과:
  - 실패 페이지로 이동한다.
  - 잔액은 증가하지 않는다.
  - 사용자에게 과도한 내부 오류 문구가 보이지 않는다.

#### CASH-CHARGE-06 성공 URL 재진입

- 사전조건: 충전 성공 직후 성공 URL 확보
- 단계:
  1. 성공 URL 새로고침
  2. 같은 URL 재호출
- 기대결과:
  - 중복 적립되지 않는다.
  - 결과 화면은 안정적으로 유지된다.

#### CASH-CHARGE-07 전화 인증/온보딩 미완료 상태

- 사전조건: 전화 인증 미완료 계정, 온보딩 미완료 계정
- 단계:
  1. `/cash/charge` 진입
  2. 충전 시도
- 기대결과:
  - 각각 `verify-phone`, `welcome`으로 유도된다.

#### CASH-CHARGE-08 충전 중 브라우저 이탈

- 사전조건: 동일
- 단계:
  1. 주문 생성 후 결제 완료 전에 탭 종료 또는 이동
  2. 다시 `/cash/charge` 진입
- 기대결과:
  - 잔액은 변하지 않는다.
  - 최근 주문 상태가 운영상 확인 가능하다.

  1. 다시 신청 시도

  - 이미 신청됨 안내
  - 추가 차감 없음

#### MATCH-APPLY-05 정원 마감 매치

- 사전조건: `M3`
- 단계:
  1. 신청 시도
- 기대결과:
  - 마감 오류
  - 차감 없음

#### MATCH-APPLY-06 시작 임박 또는 시작 후 매치

  - 차감도 1회만 발생

## Batch 5. 사용자 취소와 매치 환불

### 코드 감사 포인트

- 시간대별 환불 규칙, 예외 환불 모드, 쿠폰 복구, 알림/리마인더 취소가 함께 정합하게 동작하는지

### 시나리오

- [ ] MATCH-CANCEL-01 매치 2일 전 취소 시 무료 취소가 적용된다.
- [ ] MATCH-CANCEL-02 매치 1일 전 취소 시 정책 환불률이 적용된다.
- [ ] MATCH-CANCEL-03 당일 2시간 전 취소 시 정책 환불률이 적용된다.
- [ ] MATCH-CANCEL-04 시작 2시간 이내 취소 시 환불 불가가 적용된다.
- [ ] MATCH-CANCEL-05 참가자 미달 전날 예외 모드에서 취소 시 전액 환불된다.
- [ ] MATCH-CANCEL-06 참가자 미달 당일 예외 모드에서 취소 시 전액 환불된다.
- [ ] MATCH-CANCEL-07 강수 안내를 받은 매치 취소 시 예외 환불이 적용된다.
- [ ] MATCH-CANCEL-08 쿠폰 사용 신청 취소 시 쿠폰 복구 여부가 정책과 맞는다.
- [ ] MATCH-CANCEL-09 취소 후 리마인더가 취소되고 사용자 취소 알림이 발송된다.

## Batch 6. 캐시 환불

### 코드 감사 포인트

- 환불 신청 API 입력 검증, 중복 신청 방지, 관리자 승인/반려 후 ledger 정합성
- 환불 알림 변수와 실제 표시 값이 템플릿과 맞는지

### 시나리오

#### CASH-REFUND-01 정상 환불 신청

- 사전조건: 환불 가능한 캐시 보유
- 단계:
  1. `/mypage/cash` 진입
  2. 환불 신청
  3. 은행/계좌/예금주 입력
  4. 일정 동의 후 제출
- 기대결과:
  - 대기 상태 표시
  - 보류 거래가 내역에 반영된다.

#### CASH-REFUND-02 입력 검증

- 사전조건: 동일
- 단계:
  1. 은행 미선택
  2. 계좌번호 잘못 입력
  3. 예금주 미입력
  4. 일정 동의 미체크
- 기대결과:
  - 각 입력 오류가 분리돼 표시된다.

#### CASH-REFUND-03 중복 환불 신청

- 사전조건: 이미 대기 중인 환불 요청 존재
- 단계:
  1. 다시 신청 시도
- 기대결과:
  - 중복 신청 방지 메시지

#### CASH-REFUND-04 관리자 승인

- 사전조건: 대기 환불 요청 존재, `A1`
- 단계:
  1. 관리자 캐시 화면 진입
  2. 해당 요청 승인
- 기대결과:
  - 요청 상태가 `processed`
  - 사용자 잔액 감소 유지
  - 사용자에게 환불 완료 알림 발송

#### CASH-REFUND-05 관리자 반려

- 사전조건: 대기 환불 요청 존재, `A1`
- 단계:
  1. 요청 반려
- 기대결과:
  - 요청 상태가 `rejected`
  - 보류 금액 복구
  - 사용자 내역과 잔액이 복원

## Batch 7. 회원 탈퇴와 30일 재가입 제한

### 코드 감사 포인트

- 탈퇴 요청 시 예정 매치, 대기 충전 주문, 환불 요청이 차단 조건으로 정확히 동작하는지
- 환불 연계 탈퇴 승인/반려 후 로그인 상태와 account status가 일관적인지
- 탈퇴 후 휴대폰 재가입 차단이 30일 동안 유지되는지

### 시나리오

- [ ] WITHDRAW-01 캐시 0원, 예정 매치 0건 상태에서 탈퇴하면 즉시 탈퇴 완료 또는 대기 상태로 정상 전이한다.
- [ ] WITHDRAW-02 보유 캐시가 있으면 환불 계좌 입력과 함께 탈퇴가 진행된다.
- [ ] WITHDRAW-03 예정 매치가 있으면 탈퇴가 차단된다.
- [ ] WITHDRAW-04 미완료 충전 주문이 있으면 탈퇴가 차단된다.
- [ ] WITHDRAW-05 탈퇴 직후 기존 계정으로 재로그인 시 로그인 차단 또는 대기 안내가 나온다.
- [ ] WITHDRAW-06 같은 휴대폰 번호로 30일 이내 재가입 시 차단된다.
- [ ] WITHDRAW-07 탈퇴 연계 환불 반려 시 계정이 정상 복귀되고 로그인 가능해진다.

## Batch 8. 관리자 운영

### 코드 감사 포인트

- 관리자 취소/환불/예외 모드가 사용자 화면과 ledger에 즉시 일관되게 반영되는지
- 강수/참가자 미달 운영 액션이 중복 발송 또는 중복 환불을 만들지 않는지

### 시나리오

- [ ] ADMIN-MATCH-01 관리자 매치 생성 후 공개 홈에 정상 노출된다.
- [ ] ADMIN-MATCH-02 관리자 매치 수정 후 상세 정보가 갱신된다.
- [ ] ADMIN-MATCH-03 관리자 취소 시 신청자 전원 환불 처리된다.
- [ ] ADMIN-MATCH-04 참가자 미달 안내를 전날 기준으로 발송한다.
- [ ] ADMIN-MATCH-05 참가자 미달 안내를 당일 기준으로 발송한다.
- [ ] ADMIN-MATCH-06 강수 안내를 발송한다.
- [ ] ADMIN-MATCH-07 강수 변동 안내를 발송한다.
- [ ] ADMIN-MATCH-08 강수 취소 또는 현장 환급이 사용자 내역에 반영된다.
- [ ] ADMIN-CASH-01 수동 캐시 조정 후 사용자 잔액/내역이 일치한다.

## Batch 9. 알림, 재진입, 안정성

### 코드 감사 포인트

- 알림 발송 실패가 핵심 비즈니스 흐름을 막지 않는지
- dedupe key로 중복 발송이 방지되는지
- 새로고침/뒤로가기/재진입 시 동일 action이 중복 수행되지 않는지

### 시나리오

- [ ] NOTI-01 휴대폰 인증 SMS가 정상 발송된다.
- [ ] NOTI-02 매치 신청 알림이 정상 발송된다.
- [ ] NOTI-03 매치 취소 알림이 정상 발송된다.
- [ ] NOTI-04 캐시 충전 알림이 정상 발송된다.
- [ ] NOTI-05 캐시 환불 알림이 정상 발송된다.
- [ ] NOTI-06 템플릿 미설정 또는 발송 실패 상황에서도 신청/충전/환불 자체는 성공 처리된다.
- [ ] RESILIENCE-01 브라우저 새로고침 후 성공 화면 재진입 시 중복 반영이 없다.
- [ ] RESILIENCE-02 뒤로가기 후 같은 액션을 다시 눌러도 중복 반영이 없다.
- [ ] RESILIENCE-03 탭 복귀 또는 네트워크 지연 상황에서도 최종 상태가 한 번만 반영된다.
